# BTC SHORT, force-close 15:00 UTC, TP −$500, no stop

**Date:** 2026-06-20
**Script:** `scripts/run-btc-short-0000-tp500-backtest.ts`
**Window:** 365 days · `1h` · $1000 fixed/trade · fee 0.05%/side · TP −$500
**Context:** continuing the BTC-short clock sweep — user moved the force-close to **15:00 UTC** (much
longer hold). Tested with both 00:00 and 02:00 entry.

## Command
```bash
scripts/run-btc-short-0000-tp500-backtest.ts 365 0.05 1000 500 15 0   # entry 00:00
scripts/run-btc-short-0000-tp500-backtest.ts 365 0.05 1000 500 15 2   # entry 02:00
```

## Takeaway

Holding to 15:00 lets the $500 TP fill on more days (64% vs 52% — a longer window has more chance to
drop $500). **Entry 02:00 → exit 15:00 gives the highest gross of any short variant tested (+$146.62)**
and the least-bad net (−$217).

**Still net-negative — fees still win.** Best gross +$147 / 365 / $1000 = **0.040%/trade**, below the
0.1% round-trip fee floor. The no-stop structure stays ugly: only ~15% of forced closes are green.

**Conclusion holds across every variant tried** (entry 00/02, exit 04/08/12/15, ± 1D bear filter): the
BTC clock-short has a small positive gross edge that is *always* under the fee hurdle → not tradeable.
Root cause unchanged: BTC's intraday % edge (~0.02–0.04%/trade) sits below the 0.1% fee, and no
entry/exit-hour tuning clears it. The only net-positive clock trade found remains long SOL/BNB
(00:00→04:00, 1D-bull filter), which have the larger % range BTC lacks.
